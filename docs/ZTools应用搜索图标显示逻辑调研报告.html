一、一句话结论

ZTools 不预先抓取并存储应用图标文件。扫描时只把应用路径编码成 ztools-icon://<URL编码的应用路径> 字符串写入 Command.icon；搜索结果用普通 <img> 标签引用该伪协议 URL，<img> 触发请求时由主进程注册的自定义协议 handler 按需、串行、带 LRU 缓存地调用 C++ 原生模块从系统提取 PNG 图标返回。

这是一套"懒加载 协议流式 内存缓存 原生提取"的方案，核心文件：commandScanner/*Scanner.ts、core/iconProtocol.ts、core/native/index.ts、components/common/VerticalList.vue。

二、端到端数据流总览

扫描系统应用 生成 Command.icon = ztools-icon://<path> 存入搜索索引 搜索结果 <img src> 协议 handler 收到请求 LRU 缓存命中？ 串行队列提取 C++ 原生 getFileIcon PNG Response + 写缓存

关键点：图标 PNG 数据从不持久化到磁盘，仅以 Buffer 形式驻留在主进程内存的 LRU Map 中（上限 128 条）。

三、第 1 步：扫描阶段——为指令生成 icon 字段

3.1 自定义协议 ztools-icon://

所有平台的应用图标都统一编码为同一种伪协议 URL：

// windowsScanner.ts / macScanner.ts
export function getIconUrl(appPath: string): string {
  return `ztools-icon://${encodeURIComponent(appPath)}`
}

注意 双重编码设计：应用绝对路径（可能含中文、空格、反斜杠）先 encodeURIComponent，再作为 ztools-icon:// 的 host/path 部分。handler 侧用 decodeURIComponent 还原原始路径。

这个协议在 app.ready 之前注册为特权协议（iconProtocol.ts → registerIconScheme）：

protocol.registerSchemesAsPrivileged([{
  scheme: 'ztools-icon',
  privileges: {
    bypassCSP: true,        // 绕过 CSP，允许 <img> 加载
    secure: true,           // 视为安全来源
    supportFetchAPI: true,  // 可被 fetch
    standard: false,
    corsEnabled: false,
    stream: false
  }
}])

之后在主窗口 session（及内置插件 session）中注册 handler：registerIconProtocolForSession(session)。注释明确：外部插件 session 不注册，第三方插件访问不到应用图标。

3.2 Windows 扫描器（windowsScanner.ts）

扫描范围：开始菜单（用户 + 系统）+ 桌面，递归 .lnk / .url 快捷方式。图标路径的选择策略：

快捷方式类型icon 字段指向说明
.lnk.lnk 文件自身路径原生 SHGetFileInfoW 能解析快捷方式图标（含自定义图标），无需先 resolve 目标
.url（应用协议，如 steam://）优先 .url 内的 IconFile，否则 .url 自身解析 [InternetShortcut] 段的 IconFile= 字段
.lnk 指向 .url同上，按 .url 处理递归解析
// 始终用 .lnk 自身路径作为图标来源
const icon = getIconUrl(fullPath)
app = { name: appName, path: fullPath, icon, acronym, _dedupeTarget: targetPath }

附带机制：本地化显示名（解决磁盘文件名是英文 File Explorer.lnk 但应显示"文件资源管理器"）——通过读 desktop.ini 的 [LocalizedFileNames] 段 + 原生 MuiResolver 解析 @dll,-id 资源引用。这是名称而非图标的处理，但与图标归属同一 Command 对象。

3.3 macOS 扫描器（macScanner.ts）

扫描范围：/Applications、/System/Applications、/System/Applications/Utilities/、~/Applications，只取顶层 .app 目录。

// 应用直接用 .app 路径交给原生层提取图标
const iconUrl = `ztools-icon://${encodeURIComponent(appPath)}`
return { name, path: appPath, icon: iconUrl, aliases, acronym }

macOS 不需要像 Windows 那样选 .lnk/IconFile，因为 .app bundle 路径本身就足够让原生层（AppKit NSWorkspace）拿到图标。扫描使用 pLimit(tasks, 50) 并发控制读 plist 名称，但图标此时完全不提取。

四、第 2 步：渲染阶段——搜索结果加载图标

搜索结果列表由 components/common/VerticalList.vue 渲染，图标就是一个普通 <img> 标签：

<div class="item-icon">
  <img v-if="app.icon" :src="app.icon" alt="" draggable="false" />
  <span v-if="isDevelopmentPluginName(app.pluginName)" class="item-dev-badge">DEV</span>
</div>
.item-icon { width: 32px; height: 32px; }
.item-icon img { width: 100%; height: 100%; object-fit: contain; }

当 app.icon = "ztools-icon://C%3A%2F...notepad.exe" 时，浏览器内核解析 <img src> 会向该自定义协议发请求，触发主进程注册的 handler。object-fit: contain 保证不同尺寸图标统一在 32×32 框内居中且不变形。

重要区分：搜索结果用的是普通 <img>，不是 AdaptiveIcon.vue。也就是说应用图标不做"自适应反色"处理（详见第八节）。AdaptiveIcon 用于其它需要根据深浅色模式自动反色的场景（如某些插件图标）。

五、第 3 步：协议 Handler——缓存与提取

iconProtocol.ts 中 handler 的核心逻辑：

session.protocol.handle('ztools-icon', async (request) => {
  const iconPath = decodeURIComponent(request.url.replace('ztools-icon://', ''))

  // 1. 命中内存缓存：刷新 LRU 顺序后直接返回
  const cached = iconMemoryCache.get(iconPath)
  if (cached) { setIconCache(iconPath, cached); return createIconResponse(cached) }

  // 2. 未命中：串行队列提取
  const buffer = await extractIconQueued(iconPath)
  setIconCache(iconPath, buffer)
  return createIconResponse(buffer)
})

LRU 内存缓存

  • 数据结构：Map<string, Buffer>，利用 Map 按插入顺序迭代的特性实现 LRU
  • 容量上限：MAX_ICON_CACHE = 128
  • 命中时 delete 再 set，把 key 移到末尾（最近使用）；满时淘汰 keys().next().value（最久未用）

响应构造

function createIconResponse(buffer: Buffer): Response {
  return new Response(new Uint8Array(buffer), {
    status: 200,
    headers: {
      'content-type': 'image/png',
      'content-length': buffer.length.toString(),
      'access-control-allow-origin': '*'
    }
  })
}

图标统一以 PNG 格式返回（原生层负责把系统图标位图编码为 PNG Buffer）。

Base64 变体

除协议流式外，还提供 getFileIconAsBase64(path) 返回 data:image/png;base64,<...>，供需要内联 base64 的场景（如写入剪贴板、传给 analyzeImage、超级面板）使用，复用同一套缓存与队列。

六、第 4 步：原生层——C++ 图标提取

handler 最终调用 IconExtractor.getFileIcon(path)，对应 C++ 原生模块（resources/lib/{mac,win}/ztools_native.node）。

平台输入底层 API输出
Windows.exe / .lnk / .dll / 任意文件SHGetFileInfoW（解析快捷方式含自定义图标）PNG Buffer
macOS.app bundle 路径AppKit NSWorkspace 图标 APIPNG Buffer
Linux—不加载原生模块，getFileIcon 直接抛错—
// native/index.ts
export class IconExtractor {
  static getFileIcon(filePath: string): Promise<Buffer> {
    if (platform !== 'win32' && platform !== 'darwin') {
      throw new Error('getFileIcon is only supported on Windows and macOS')
    }
    return (addon as NativeAddon).getFileIcon(filePath)
  }
}

七、性能与并发设计

关键工程细节：串行提取队列

macOS AppKit 图标 API 在高并发下可能返回 null。为避免空图标，iconProtocol.ts 用 Promise 链实现了一个全局串行队列，保证同一时刻只有一个原生提取任务在执行：

let extractionQueue: Promise<void> = Promise.resolve()

function extractIconQueued(iconPath: string): Promise<Buffer> {
  const task = extractionQueue.then(() => extractIcon(iconPath))
  extractionQueue = task.then(() => undefined, () => undefined) // 错误不阻塞后续
  return task
}
  • 串行 + 缓存：首次提取慢（受限于原生 API），但缓存命中后直接返回 Buffer，零原生调用
  • 错误隔离：单个提取失败不会卡死队列（() => undefined 兜底）
  • 无磁盘 IO：图标只进内存 Map，不落盘，避免扫描期大量小文件写入
  • 渲染零成本：搜索结果列表不预加载所有图标，<img> 按可见性自然触发（注：列表本身未用 IntersectionObserver 懒加载，依赖浏览器对 <img> 的原生调度）

八、与"自适应图标"的关系（易混淆点）

仓库里还有一个 AdaptiveIcon.vue 组件和 imageAnalysis.ts 分析模块，容易误以为搜索图标也走它们。实际不是：

搜索结果应用图标AdaptiveIcon（自适应）
组件VerticalList.vue 普通 <img>AdaptiveIcon.vue
是否分析颜色否是，调 ztools.analyzeImage
反色处理无深色图标+深色模式 → filter: invert(1)

关键证据：imageAnalysis.ts 开头对 ztools-icon:// 开头的输入直接返回 needsAdaptation: false，明确跳过分析：

if (imagePath.startsWith('ztools-icon://')) {
  // 动态生成的应用图标，不做颜色分析，直接返回默认值
  return { isSimpleIcon: false, mainColor: null, isDark: false, needsAdaptation: false }
}

原因 系统应用图标由 OS 原生提取，本身已带正确透明度与配色，无需 ZTools 再做深浅色反色；自适应反色主要服务于那些"白底/黑底单色"的插件图标。AdaptiveIcon 还带 IntersectionObserver（rootMargin: 200px）懒加载与分析，是更重的组件，不适合塞进高频渲染的搜索列表。

九、边界场景与降级

  • 提取失败：handler catch 后返回 404 'Icon Error'，<img> 加载失败（VerticalList 未显式处理 @error，将显示空白或浏览器默认 broken icon）
  • macOS 并发 null：由串行队列规避；若仍 null，extractIcon 抛 'Failed to extract icon'
  • Linux：原生模块不加载，应用图标功能不可用（getFileIcon 抛错）
  • 插件图标：走另一条路 —— common/iconUtils.ts → normalizeIconPath 把插件 plugin.json 里的 logo（相对路径）转成 file:/// 绝对 URL，或保留 data: / http(s)://，不经 ztools-icon://
  • 外部插件隔离：registerIconProtocolForSession 只对内置插件 session 注册 handler，第三方插件无法读取宿主应用图标（安全隔离）

十、设计要点小结

  1. 路径即图标：扫描期只存 ztools-icon://<encoded path> 字符串，零图标 IO，扫描极快。
  2. 懒加载 + 协议流：图标在 <img> 实际请求时才提取，天然按需。
  3. 内存 LRU：128 条上限，命中零成本，未命中走原生；不污染磁盘。
  4. 串行队列：规避 macOS AppKit 并发返回 null 的原生缺陷，是稳定性关键。
  5. 统一 PNG：跨平台统一输出 PNG，渲染端无需关心来源格式。
  6. 职责分离：应用图标（ztools-icon://，普通 img，不反色）vs 插件图标（file:///，可走 AdaptiveIcon 反色）两套独立链路。
  7. 安全边界：协议只注册给主窗口与内置插件 session，第三方插件拿不到。
对 forestools（本项目）的借鉴：本项目是 Tauri 应用。若要实现类似"搜索应用带系统图标"，可对应映射为：① Tauri/Rust 侧扫描 .lnk/plist 生成指令列表；② 用 tauri-plugin-shell 或自写 Rust 调 SHGetFileInfo/NSWorkspace 提取图标为 PNG bytes；③ 通过自定义 tauri:// 协议或 convertFileSrc+IPC 流式返回；④ 前端 <img> 引用。ZTools 的"串行队列规避并发 null"与"内存 LRU 不落盘"两点最值得直接照搬。
